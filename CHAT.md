现在我希望对这个仓库做一个彻底的规整：
1. 把目前这些代码，挪到 web 文件夹中
2. 新增一个 server 文件夹，这个 server 文件夹底层可以对接到 sqlite 或者 postgresql 数据库，server 的目的是实现 邮箱认证登录和加密存储，最终为了实现跨平台订阅同步，我希望底层使用WebSocket，所以技术选型上，可以考虑使用 golang或者其它高性能低成本的服务。我的目的是商业化运作+支持开源自部署。免费用户就像现在这样，使用浏览器自身的存储去保存这些密钥。而使用云服务，就意味着可以托管密钥。但是自己托管服务器本身可能也是一个高度不确定的问题。服务可能会因为某种原因而损坏，从而影响用户体验。我初步的想法是优先使用 cloudflare 这种云服务，然后同时部署到自己的服务器，使用分布式的方式实现云服务和我自己服务器之间的实时同步。这里面涉及到一些技术选型的问题，我希望我们可以展开套讨论讨论，甚至 WebSocket 的技术选型可能也不是最好的。
3. 我的核心理念不是为了赚钱， 因此如果可以，我们使用 Google 账户登录或者 Github 账户登录的时候，我希望充分使用这些账号提供商所提供的能力，比如我知道的，如果使用 Github 授权登录，我希望授权 gist 权限给我，这样我就能将用户的数据存储在 gist 上，不需要再依赖我的服务器。然后用户只需要多端登录他们的 Github 账户授权，就可以进行跨设备同步，这时候用的就是轮询，默认 cache-tty 是 15 分钟。同时界面上也提供了一个sync 按钮，方便用户从 gist 上进行拉取更新。如果用了 Google 账户，是不是可以让用户授权 Google Drive，我用 GoogleDrive 来进行存储和轮询？
4. 按照这种思路，我们自己的后端，我想通过 Web3 的加密技术来实现账号的存储：用户可以填入它的任意密钥，如果识别出来不是助记词，那么就通过 sha256 就可以生成符合助记词标准的密钥。通过这个密钥，实现非对称加密，也就意味着我自己的服务器上也不会明文存储这些密钥。我个人更喜欢用户通过填入自己的助记词或者密钥，就能实现到我的服务器上进行数据下载。
5. 所以我希望实现多 Provider 模式：Github gist、Google Drive、我的密钥存储方案。
